# Plan: torrent-swarm-dashboard & torrent-swarm-status-api

## Overview

Two new components that make the Nano snapshot swarm discoverable without requiring end-users to run the Mirror client:

1. **`status-api/`** — A "dumb" Fly.io FastAPI service that receives pushed status updates from the Producer and serves them as JSON + .torrent files. No DHT, no libtorrent, no heavy runtime. Validates Ed25519 signatures against the embedded AUTHORITY_PUBKEY. Cloudflare sits in front for CDN caching.

2. **`dashboard/`** — A static GitHub Pages site (single `index.html`) that fetches from the status API and renders the current snapshot info, magnet link, .torrent download, and direct HTTP download — all SEO-optimized for search engines.

The **Producer** gets a new push step: after publishing to DHT, it POSTs the snapshot status + .torrent file + Ed25519 signature to the status API.

### 1. `status-api/app/main.py` — FastAPI Service

**Dependencies:** `fastapi`, `uvicorn`, `pynacl`, `pydantic`, `jinja2` (no libtorrent!)

**Routes:**

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `POST` | `/api/push` | Ed25519 signature | Receive status push from Producer |
| `GET`  | `/api/status` | None | JSON: current snapshot metadata + magnet + links |
| `GET`  | `/api/status-fragment` | None | HTML fragment for HTMX (CORS-enabled) |
| `GET`  | `/api/torrent` | None | Serve .torrent file (`Content-Type: application/x-bittorrent`) |
| `GET`  | `/` | None | Full HTML dashboard (SSR, SEO-friendly) |
| `GET`  | `/health` | None | Liveness probe |

**POST `/api/push` — Push Protocol**

The Producer sends a JSON payload with an Ed25519 signature over the canonical message bytes. Signature covers all fields except `signature` itself, ensuring tamper-proof updates.

Request body:
```json
{
  "sequence": 42,
  "info_hash": "924a5772b2db194d...",
  "torrent_name": "nano-ledger-snapshot.7z",
  "web_seed_url": "https://s3.us-east-2.amazonaws.com/repo.nano.org/snapshots/latest/nano-ledger-snapshot.7z",
  "piece_size": 33554432,
  "snapshot_size_bytes": 64320000000,
  "timestamp": "2026-04-23T00:00:00Z",
  "torrent_file_b64": "<base64-encoded .torrent bytes>",
  "signature": "<hex Ed25519 signature>"
}
```

**Signature scheme:**
1. Construct canonical message: `f"{sequence}:{info_hash}:{timestamp}"` (ASCII bytes)
2. Sign with Ed25519 using DHT_PRIVATE_KEY (same key used for DHT mutable items)
3. API validates signature against `AUTHORITY_PUBKEY` (embedded in Docker image, same pattern as Mirror)

The canonical message is deliberately simple and stable — covers sequence, info_hash, and timestamp. The .torrent file content is NOT in the signature (it would be redundant since the info_hash is derived from it; if info_hash matches, the .torrent is valid).

**Validation rules:**
- Reject if `sequence` < current stored sequence (replay protection)
- Reject if signature verification fails against AUTHORITY_PUBKEY
- Accept and store if `sequence` >= current

**Storage:** Write `status.json` and `torrent.bin` to `/data/` (persistent Fly.io volume). Read from file on startup so the API survives redeploys without needing a push from the Producer.

**GET `/api/status` response:**

- `Access-Control-Allow-Origin: *` (needed for GitHub Pages to fetch cross-origin)

### 2. `producer/push_status.py` — Push Script

A new module called after DHT publish in `daily-snapshot.sh`.

**Signature construction** (reuses `shared/bep46.py` signing logic):
```python
from nacl.signing import SigningKey

def sign_push(private_key_hex: str, sequence: int, info_hash: str, timestamp: str) -> str:
    seed = bytes.fromhex(private_key_hex[:64] if len(private_key_hex) > 64 else private_key_hex)
    signing_key = SigningKey(seed)
    message = f"{sequence}:{info_hash}:{timestamp}".encode("ascii")
    signed = signing_key.sign(message)
    return signed.signature.hex()
```

**Push function:**
```python
def push_status(
    status_api_url: str,
    private_key_hex: str,
    sequence: int,
    info_hash: str,
    torrent_name: str,
    web_seed_url: str,
    piece_size: int,
    snapshot_file: str,
    torrent_file: str,
) -> dict:
    # Read .torrent file, base64-encode
    torrent_bytes = Path(torrent_file).read_bytes()
    torrent_b64 = base64.b64encode(torrent_bytes).decode("ascii")

    timestamp = datetime.datetime.now(datetime.timezone.utc).isoformat()
    signature = sign_push(private_key_hex, sequence, info_hash, timestamp)

    payload = {
        "sequence": sequence,
        "info_hash": info_hash,
        "torrent_name": torrent_name,
        "web_seed_url": web_seed_url,
        "piece_size": piece_size,
        "snapshot_size_bytes": Path(snapshot_file).stat().st_size,
        "timestamp": timestamp,
        "torrent_file_b64": torrent_b64,
        "signature": signature,
    }

    resp = requests.post(f"{status_api_url}/api/push", json=payload, timeout=30)
    resp.raise_for_status()
    return resp.json()
```

**Integration into `daily-snapshot.sh`:**

After the existing Step 5 (publish to DHT) and Step 6 (restart seeder), add:

### 4. `dashboard/index.html` — GitHub Pages + HTMX

A single self-contained HTML file using **HTMX** for dynamic updates. No custom JS fetch logic — HTMX handles polling and DOM swapping.

**How HTMX works here:**
- The page loads with skeleton/placeholder content
- HTMX polls `/api/status-fragment` every 60 seconds
- The API returns an HTML fragment (just the dynamic `<section>` tags)
- HTMX swaps the fragment into the page, replacing the skeleton

This means:
- Zero custom JavaScript — just HTMX attributes in the HTML
- The API serves both full HTML pages (`/`) and fragments (`/api/status-fragment`)
- GitHub Pages and the API's own `/` render identically
- SEO is fully covered (initial HTML contains meta tags; dynamic content augments)

**HTML structure (both `dashboard/index.html` and API's `/`):**
```html
<!DOCTYPE html>
<html lang="en">
<head>
    <title>Nano Ledger Snapshot — P2P Bootstrap Download</title>
    <meta name="description" content="...">
    <!-- SEO meta tags, og:tags, Schema.org structured data -->
    <script src="https://unpkg.com/htmx.org@2.0.4"></script>
    <style>/* inline CSS */</style>
</head>
<body>
    <h1>Nano (XNO) Ledger Snapshot</h1>

    <!-- This section polls the API and swaps itself -->
    <div hx-get="https://nano-snapshot-hub.fly.dev/api/status-fragment"
         hx-trigger="load, every 60s"
         hx-swap="innerHTML">
        <!-- Skeleton content shown before first load -->
        <p>Loading snapshot status...</p>
    </div>

    <!-- Static sections (no HTMX needed) -->
    <section id="seed">
        <h2>Help Seed</h2>
```

The only difference between `dashboard/index.html` (GitHub Pages) and the API's `/` template is the `hx-get` URL — the GitHub Pages version points to `https://nano-snapshot-hub.fly.dev/api/status-fragment`, while the API's own `/` uses a relative URL `/api/status-fragment`.

**SEO keywords targeted:**
- "nano xno ledger snapshot"
- "nano node bootstrap"
- "nano blockchain snapshot download"
- "nano ledger p2p"

**Fallback for no-JS users:** The `<noscript>` tag shows a link to the API's `/api/status` JSON endpoint and a direct HTTP download link for the web seed.

### 8. Cloudflare Caching Strategy

Since the API sits behind Cloudflare proxy:
- `GET /api/status-fragment` → `Cache-Control: public, max-age=300` (5 min)
- `GET /api/status` → `Cache-Control: public, max-age=600` (10 min)
- `GET /api/torrent` → `Cache-Control: public, max-age=3600, immutable` (1 hour; .torrent only changes on new snapshot)
- `GET /` → `Cache-Control: public, max-age=300` (5 min)
- `POST /api/push` → No caching (always hits origin)
- CORS headers on `/api/status-fragment` and `/api/status` for cross-origin access from GitHub Pages

This means most visitors hit CF's edge cache. The Fly.io VM only processes:
- Push from Producer (~1 request/hour)
- Cache misses (~1-2/minute, depending on traffic)

Expected cost: **$2-3/month** on Fly.io (mostly idle).

### 9. Signature Verification (API Side)

```python
from nacl.signing import VerifyKey
from nacl.exceptions import BadSignatureError

def verify_push(payload: PushRequest, authority_pubkey_hex: str) -> bool:
    pubkey_bytes = bytes.fromhex(authority_pubkey_hex)
    verify_key = VerifyKey(pubkey_bytes)
    message = f"{payload.sequence}:{payload.info_hash}:{payload.timestamp}".encode("ascii")
    try:
        verify_key.verify(message, bytes.fromhex(payload.signature))
        return True
    except BadSignatureError:
        return False
```

This reuses the exact same Ed25519 key that the Mirror uses to verify DHT mutable items. The DHT_PRIVATE_KEY on the Producer side signs both the DHT item and the push payload with the same key — no new secret key needed.

## Key Design Decisions

1. **No DHT in the API** — The API is a passive receiver. All DHT intelligence stays in the Producer and Mirror. The API just stores and serves what it's told, after signature verification.

2. **Same Ed25519 key** — No new secret. The Producer's existing DHT_PRIVATE_KEY signs push payloads. The API validates with the same AUTHORITY_PUBKEY that mirrors already have baked in.

3. **Push, not pull** — The Producer pushes to the API after each publish. No need for the API to open ports to DHT or maintain long-lived sessions. The push is triggered both by the hourly systemd timer AND immediately after `daily-snapshot.sh` completes.

4. **Cloudflare caching** — The .torrent file (~100-200KB) rarely changes. CF caches it for 1 hour. The status JSON changes only on new snapshots. Most traffic never hits the Fly.io VM.

5. **Persistent volume on Fly.io** — `/data` mounted as a 1GB volume stores `status.json` and `torrent.bin`. Survives VM restarts and deploys. No database needed.

6. **Magnet link construction** — Built deterministically from info_hash + torrent_name + tracker list. No dynamic DHT needed. Uses `urn:btmh:` (BitTorrent v2 Merkle hash format).

7. **SSR fallback + HTMX** — The API serves its own `/` as a full HTML page rendered with Jinja2 (for SEO and users who find the API directly). The dynamic section is loaded via HTMX (`hx-get="/api/status-fragment"`). GitHub Pages uses the same HTML but with an absolute API URL. Both share the same visual design and HTMX-driven updates.

8. **Non-fatal push failures** — If the API is down or unreachable, the push script logs a warning and exits 0. DHT publishing (the important part) has already succeeded. The status push is best-effort.
