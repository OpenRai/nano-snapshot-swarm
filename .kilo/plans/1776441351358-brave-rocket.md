# Plan: Replace cron with systemd timer for daily-snapshot pipeline

## Why

- Cron sends mail on every run (even successful ones produce stdout via `tee -a`)
- No one wants 10,458 cron mail messages from `openrai`
- systemd timer + service = journald logging, no mail, proper status, `journalctl -u nano-snapshot` to debug
- The script already uses `tee -a` for its own log file â€” journald captures stdout/stderr on top of that

## What to do

### 1. Remove the crontab entry

```bash
ssh bandwidth-martyr "crontab -r"
```

### 2. Create systemd service unit

File: `/etc/systemd/system/nano-snapshot.service`

```ini
[Unit]
Description=Nano P2P Snapshot Pipeline (daily fetch, compact, compress, publish)
After=network-online.target
Wants=network-online.target

[Service]
Type=oneshot
User=openrai
Environment=OUTPUT_DIR=/opt/nano-snapshots
Environment=WEB_SEED_URL=https://s3.us-east-2.amazonaws.com/repo.nano.org/snapshots/latest
Environment=DHT_PRIVATE_KEY=<from /home/openrai/.env>
Environment=AUTHORITY_PUBKEY=<from /home/openrai/.env>
Environment=LOG_FILE=/opt/nano-snapshots/nano-snapshot.log
ExecStart=/opt/nano-bootstrap-swarm/scripts/daily-snapshot.sh
StandardOutput=journal
StandardError=journal
SyslogIdentifier=nano-snapshot
TimeoutStopSec=3600
```

Notes:
- `Type=oneshot` â€” runs once, exits. The timer calls it.
- `User=openrai` â€” runs as the service user, like cron would.
- Environment vars inline so we don't need to `source ~/.env` from the script. But the script currently does `source /home/openrai/.env` â€” we should make that optional (skip if vars already set).
- `TimeoutStopSec=3600` â€” 56 GB download can take hours, give it time.
- `StandardOutput=journal` + `StandardError=journal` â€” all output goes to journald.

### 3. Create systemd timer unit

File: `/etc/systemd/system/nano-snapshot.timer`

```ini
[Unit]
Description=Run Nano P2P Snapshot Pipeline daily at 02:00 UTC

[Timer]
OnCalendar=*-*-* 02:00:00 UTC
Persistent=true
RandomizedDelaySec=300

[Install]
WantedBy=timers.target
```

Notes:
- `Persistent=true` â€” if the server was down at 02:00, run it on next boot.
- `RandomizedDelaySec=300` â€” jitter up to 5 minutes to avoid thundering herd with other Nano producers (not that there are any, but good practice).

### 4. Modify `daily-snapshot.sh` to make `source ~/.env` optional

The script currently does `source /home/openrai/.env` unconditionally. Under systemd, env vars come from the service unit's `Environment=` directives. Change to:

```bash
# Load env file only if vars not already set (e.g., from systemd)
if [ -z "$DHT_PRIVATE_KEY" ] && [ -f /home/openrai/.env ]; then
    source /home/openrai/.env
fi
```

This way it works both from systemd (env vars from unit file) and manually (env vars from `.env`).

### 5. Enable and start the timer

```bash
ssh root@185.208.206.54 "systemctl daemon-reload && systemctl enable nano-snapshot.timer && systemctl start nano-snapshot.timer"
```

### 6. Verify

```bash
systemctl status nano-snapshot.timer
systemctl list-timers nano-snapshot
journalctl -u nano-snapshot --no-pager -n 20
```

### 7. Test a manual run

```bash
ssh bandwidth-martyr "sudo systemctl start nano-snapshot.service"
# Then watch:
ssh bandwidth-martyr "sudo journalctl -u nano-snapshot -f"
```

## Service unit design decisions

- **Env vars in unit file vs `.env` file**: Using `EnvironmentFile=/home/openrai/.env` would be cleaner but requires the `.env` to use `KEY=VALUE` format without `export` prefix, which conflicts with sourcing it in shell. Since the script already handles both cases with the conditional `source`, putting them in `Environment=` directives in the unit file is explicit and avoids format conflicts.
  
  Actually â€” even simpler: just use `EnvironmentFile=` with a separate plain file, OR keep the conditional source in the script. The conditional source approach means we don't duplicate the keys in the unit file and the `.env` remains the single source of truth.

  **Revised approach**: Use `EnvironmentFile=-/home/openrai/.env` in the unit (the `-` prefix makes it non-fatal if missing), and make the script's `source` conditional. The `.env` uses `export KEY=VALUE` format which systemd `EnvironmentFile` ignores the `export` keyword and just parses `KEY=VALUE`. Wait â€” actually systemd `EnvironmentFile` does NOT support `export` prefix. It expects plain `KEY=VALUE` lines.

  **Final decision**: Change `/home/openrai/.env` to use plain `KEY=VALUE` format (no `export`). Both `source` in bash and systemd `EnvironmentFile` will work. Then use `EnvironmentFile=` in the unit.

## Updated .env format

Current format:
```bash
export DHT_PRIVATE_KEY=cdc5...
export AUTHORITY_PUBKEY=eb63...
```

New format (works with both bash `source` and systemd `EnvironmentFile`):
```bash
DHT_PRIVATE_KEY=cdc5...
AUTHORITY_PUBKEY=eb63...
NANO_LEDGER_PATH=/var/nano/data/data.ldb
OUTPUT_DIR=/opt/nano-snapshots
```

`source` in bash still works fine without `export` â€” the vars are set in the shell. For systemd, `EnvironmentFile` parses `KEY=VALUE` directly.
